> **Roles:** Principal (S6) — full builder · MIS Coordinator (S4) — build and save reports · Academic Coordinator (S4) — academic data reports only · Accounts Officer (S3) — financial data reports only

---

## 1. Purpose

Not every management question fits a pre-built report. The Custom Report Builder allows authorised users to build ad-hoc reports by selecting data domains, filters, groupings, and output formats. It provides self-service analytics without requiring developer involvement for every one-off management query.

Use cases:
- "How many students in Classes IX–X have >90% attendance AND >75% marks in all subjects?" (for merit scholarship shortlisting)
- "Show all staff who have more than 8 CL days in the last 6 months" (leave pattern review)
- "List all parents who have not attended any PTM in the last 2 years" (engagement review)
- "All students with a fee concession AND below-75% attendance" (welfare cross-check)

## 2. Report Builder Interface

```
CUSTOM REPORT BUILDER

Step 1: Choose Data Domain(s)
  ○ Students (A-series: profile, class, section, admission)
  ○ Attendance (E-01: student daily attendance)
  ● Academic Performance (B-series: marks, grades, subject-wise)
  ○ Fee & Finance (D-series: fee paid, outstanding, concession)
  ○ Staff (L-series: attendance, leave, payroll — restricted roles)
  ○ Transport (I-series: student transport enrolment, route)
  ○ Welfare (J-series: counselling flag count only — no case details)
  ○ Compliance (K-series: item status, scores)

Step 2: Select Fields
  Available fields (Academic domain selected):
  ✅ Student name
  ✅ Student ID
  ✅ Class / Section
  ✅ Roll number
  ✅ [Subject] Score — select subject: Mathematics
  ✅ [Subject] Score — English
  ☐ Guardian name (add)
  ☐ Contact number (add — if guardian contact needed)
  ☐ Attendance %

Step 3: Apply Filters
  Filter 1: Class = IX OR X (multi-select)
  Filter 2: Mathematics score ≥ 75%
  Filter 3: English score ≥ 75%
  [+ Add Filter]

Step 4: Group By / Sort By
  Group by: Class, then Section
  Sort by: Mathematics score (descending)

Step 5: Output Format
  ● Table (on screen)
  ○ Export CSV
  ○ Export PDF
  ○ Save as named report (rerun periodically)

[Run Report]  [Save as: "Merit Scholarship Candidates Q1 2026"]
```

## 5. API Endpoints

| # | Method | Endpoint | Description |
|---|---|---|---|
| 1 | `POST` | `/api/v1/school/{id}/mis/custom/run/` | Run ad-hoc report with payload |
| 2 | `POST` | `/api/v1/school/{id}/mis/custom/save/` | Save named report definition |
| 3 | `GET` | `/api/v1/school/{id}/mis/custom/saved/` | List saved reports |
| 4 | `POST` | `/api/v1/school/{id}/mis/custom/saved/{report_id}/run/` | Run saved report |
| 5 | `GET` | `/api/v1/school/{id}/mis/custom/saved/{report_id}/export/?fmt=csv` | Export result |
| 6 | `GET` | `/api/v1/school/{id}/mis/custom/schema/` | Available fields per domain |

## 6. Business Rules

- Custom reports respect the same role-based access as underlying modules — if a user cannot see individual payroll data, they cannot include salary fields in a custom report; the schema API (endpoint 6) returns only fields the requesting user is authorised to access
- POCSO and welfare case details are never available as fields in the custom report builder — only anonymised counts; this is a hard restriction regardless of role
- Reports that include contact numbers (phone, email) are flagged under DPDPA purpose limitation — the system asks the user to confirm the purpose before including contact data in a report (e.g., "I am generating this for the purpose of scholarship communication — confirmed"); this creates an audit log of data use
- Saved reports with personal data are auto-expired after 30 days from last run; the report definition (query) remains; only the cached result is purged; users must re-run to get fresh data
- Reports involving student PII exported to CSV require Principal authorisation or belong to a Principal-delegated coordinator; the system logs who exported what and when; this is DPDPA audit compliance for data use
