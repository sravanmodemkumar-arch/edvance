## 4. API Endpoints

| # | Method | Endpoint | Description |
|---|--------|----------|-------------|
| 1 | `GET` | `/api/v1/parent/dashboard/overview` | Unified snapshot for all linked children (attendance, last result, fee status, alerts) |
| 2 | `GET` | `/api/v1/parent/dashboard/child/{link_id}` | Detailed view for a single child at a specific institution |
| 3 | `GET` | `/api/v1/parent/dashboard/child/{link_id}/attendance` | Full attendance calendar with day-wise status and absence reasons |
| 4 | `GET` | `/api/v1/parent/dashboard/child/{link_id}/results` | All exam/test results with marks, rank, and subject-wise breakdown |
| 5 | `GET` | `/api/v1/parent/dashboard/child/{link_id}/fees` | Fee summary — term-wise paid/due status, receipts, deadlines |
| 6 | `GET` | `/api/v1/parent/dashboard/child/{link_id}/upcoming` | Upcoming events — PTMs, exams, fee deadlines, institutional notices |
| 7 | `GET` | `/api/v1/parent/dashboard/alerts` | Aggregated alerts across all children, sorted by priority and recency |
| 8 | `POST` | `/api/v1/parent/dashboard/child/{link_id}/rsvp` | RSVP to events (PTMs, open days) with attending/not-attending status |

## 5. Business Rules

- **BR-01 — Dashboard Data Aggregation and Caching Strategy:** The unified dashboard fetches data from multiple institution backends in parallel and assembles a composite view for the parent. Since each institution operates as a separate tenant with its own database shard, the system uses an API gateway that fans out requests to each linked institution's API and merges responses with a maximum timeout of 3 seconds per institution. If an institution's API is temporarily unreachable (e.g., Sri Chaitanya's server is undergoing maintenance), the dashboard gracefully degrades by showing the last cached snapshot for that child with a "Last updated X hours ago" label, rather than failing the entire dashboard. Cached snapshots are stored in Redis with a 4-hour TTL and are refreshed on each successful API call. The parent never sees a completely blank dashboard even if all institutions are temporarily down, because the initial linking process captures a baseline snapshot that serves as the fallback of last resort.

- **BR-02 — Child Switcher Disambiguation for Same-Name Children:** When the same child is enrolled at multiple institutions (as is common in India where a college student simultaneously attends coaching for competitive exams), the child switcher must always display the institution name alongside the child's name to prevent confusion. The tab label format is "[Child Name] — [Institution Short Name], [City]" on desktop and a truncated version on mobile. The system detects potential same-name collisions during the linking process by comparing the child's name (case-insensitive, ignoring middle names) and date of birth across all existing links for that parent account. If a collision is detected, the system proactively asks the parent during linking whether this is the same child at a new institution or a different child with the same name (e.g., cousins). This classification determines whether the children are grouped together in analytics views or treated as separate individuals.

- **BR-03 — Real-Time Attendance Notification Push:** When a child is marked absent at any linked institution, the parent receives an immediate push notification via the configured channel (SMS, WhatsApp, or app notification, in order of parent preference). For Mrs. Lakshmi Devi, if Priya is marked absent at Sri Chaitanya at 8:30 AM, the notification is dispatched within 60 seconds of the attendance being recorded by the teacher. The notification includes the child's name, institution, date, and period/class missed. To prevent notification fatigue, the system batches multiple absence events from the same day into a single summary notification sent at 11:00 AM if the child has been marked absent in multiple periods. Parents can configure a "silent hours" window (default 9 PM to 7 AM) during which non-urgent notifications are queued and delivered at 7 AM the next morning. Urgent notifications such as fee payment failures or disciplinary actions bypass silent hours.

- **BR-04 — Cross-Institution Academic Comparison Is Explicitly Disabled:** The dashboard deliberately does not compare academic performance across different institutions or different children, because such comparisons are pedagogically harmful and contextually meaningless (comparing a B.Tech CGPA at GCEH with a Class 11 percentage at Sri Chaitanya serves no valid purpose). Each child's academic section operates within its own institutional context, showing ranks only within their class or batch, not across siblings. The system does not compute or display any "family average" or "sibling comparison" metrics. If a parent attempts to generate a combined report, the system shows a notice explaining that cross-institutional comparison is not supported because grading systems, syllabi, and evaluation methodologies differ fundamentally between institutions. This design decision was made after consultation with child psychologists and CBSE guidelines on reducing unhealthy academic competition among siblings.
