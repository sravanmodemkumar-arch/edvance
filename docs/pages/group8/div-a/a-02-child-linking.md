## 4. API Endpoints

| # | Method | Endpoint | Description |
|---|--------|----------|-------------|
| 1 | `POST` | `/api/v1/parent/link-child/verify-code` | Validate 6-digit linking code; return child + institution preview |
| 2 | `POST` | `/api/v1/parent/link-child/confirm` | Confirm link after DOB verification; creates parent_child_link record |
| 3 | `GET` | `/api/v1/parent/children` | List all linked children with institution details and link status |
| 4 | `DELETE` | `/api/v1/parent/children/{link_id}` | Unlink a child (requires re-verification; institution admin notified) |
| 5 | `POST` | `/api/v1/parent/link-child/request-code` | Request institution to generate a linking code; sends SMS to admin |
| 6 | `GET` | `/api/v1/parent/children/{link_id}/status` | Check link health — active, suspended by institution, or expired |
| 7 | `POST` | `/api/v1/institution/admin/generate-linking-code` | Institution admin generates a 6-digit code for a student-parent pair |
| 8 | `GET` | `/api/v1/institution/admin/linking-codes` | List all generated codes with status (used, expired, pending) |

## 5. Business Rules

- **BR-01 — Linking Code Generation and Expiry:** Linking codes are 6-digit numeric codes generated by the institution admin from the student management module. Each code is unique within the institution's namespace and is cryptographically generated to prevent sequential guessing (e.g., codes like 000001 through 000010 are never issued). A code is valid for exactly 72 hours from the moment of generation, after which it expires and cannot be used. If a parent does not use the code within this window, the institution admin must generate a new one. Each student can have at most 2 active (unused) linking codes at any time — one for the primary parent and one for a secondary guardian. Once a code is successfully used, it is permanently invalidated and cannot be reused. The institution receives a real-time webhook notification when a parent successfully links using their code, and the student's profile in the institution portal displays the linked parent's name and mobile number.

- **BR-02 — One Child, Multiple Institutions, Separate Links:** A single child (identified by their name and date of birth, or Aadhaar hash if available) can be linked to the same parent account through multiple institutions simultaneously. In Mrs. Lakshmi Devi's case, Ravi Kumar appears twice in her linked children list — once via Godavari College of Engineering (his B.Tech degree programme) and once via TopRank Academy (his GATE coaching test series). These are treated as completely independent institutional links with separate attendance records, fee structures, academic calendars, and notification streams. The parent dashboard aggregates data from all links but clearly labels which institution each data point originates from. If the child leaves one institution (e.g., completes the GATE test series), that specific link is marked as "Completed" but remains visible in the parent's history for record-keeping purposes.

- **BR-03 — Date-of-Birth Verification as Anti-Fraud Measure:** When a parent enters a valid linking code, the system displays the child's name and institution but requires the parent to correctly enter the child's date of birth before the link is established. This two-factor approach (possession of code + knowledge of DOB) prevents scenarios where a linking code is accidentally or maliciously shared with an unrelated person. The parent gets a maximum of 3 attempts to enter the correct DOB; after 3 failures, the code is invalidated, the institution admin is notified via SMS of a potential unauthorized linking attempt, and the parent must request a new code. The DOB check uses exact match on day, month, and year as stored in the institution's student records. In edge cases where the institution has an incorrect DOB on file, the parent must contact the institution to correct the student record before linking can succeed.

- **BR-04 — Institution Admin Code Management and Audit Trail:** Every linking code generation, usage, and expiry event is recorded in an immutable audit log accessible to both the institution admin and the EduForge platform super-admin. The audit trail captures the admin who generated the code, the timestamp, the student it was generated for, the parent who used it, and the IP address and device fingerprint of the linking request. Institution admins can view a dashboard showing how many parents have successfully linked versus how many codes are pending or expired, enabling them to follow up with parents who have not yet onboarded. Bulk code generation is supported — an admin can generate codes for an entire class section (e.g., all 60 students in CSE-B at GCEH) in a single action, and the system sends individual SMS messages to each student's registered parent mobile number containing the code and a direct link to the registration page. The bulk generation respects SMS rate limits and queues messages through the platform's SMS gateway with delivery receipts tracked.
